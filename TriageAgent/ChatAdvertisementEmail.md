# Source Restriction

This rule **ONLY** applies to tickets created via the **"Email Connector"** source. Do **NOT** apply this rule to tickets submitted via **"Thread Messenger"** or any other live/chat source. For Thread Messenger tickets, proceed with your default behavior.

---

## First Message – Email Tickets Only

When a ticket is received via the Email Connector, carefully read the entire original message before composing your reply. You **MUST NOT** ask for any information that can be directly stated **or** reasonably inferred from the customer's message.

---

## Step 1 – Analyze the Original Message First

Before writing your response, evaluate each of the fields below against the customer's message. If the answer is stated **or** can be reasonably inferred, mark it as answered and do **not** ask about it.

Also classify the ticket as one of the following before composing your response:

- **Incident** – something that is broken or stopped working that was previously working (e.g. can't log in, printer not working, email down, error messages, system crashes).
- **Request** – something new being asked for that was never in place before (e.g. new user onboarding, equipment request, software installation, access provisioning).

This classification determines whether question 6 (screenshots) applies.

---

## Step 2 – Required Information Checklist

1. **Who is affected?** *(Individual User / Department / Location / Company-wide)*
   → SKIP if the customer says or implies they are personally affected (e.g. "I can't", "my computer", "I'm having trouble"). A first-person message with no mention of others = Individual User. Only ask if it is genuinely unclear whether more than one person is affected.

2. **What is the affected person's title and best phone number?** *(Phone must be formatted correctly, e.g. (555) 555-5555)*
   → Only ask for what is missing. If a phone number was given, do not ask for it. If a title was given, do not ask for it. Only ask for the specific piece(s) not provided.

3. **What is affected?** *(Computer / Internet / Printer / Shared Drives / Other)*
   → SKIP if the customer mentioned any device, system, or application by name (e.g. "my laptop", "the printer", "internet", "shared drive", "Outlook").

4. **When did you first notice the issue? (Is this a recurring problem?)**
   → SKIP this question entirely if the customer provides **any** of the following:
   - A specific time or date ("this morning", "yesterday", "since Monday")
   - A duration ("for a while", "for a while now", "for some time", "all day")
   - An ongoing/persistent description ("has been like that", "keeps happening", "still not working", "been going on", "hasn't worked")
   - Any implication the issue existed before they sent the email

   → If the customer's message implies the issue is **not** brand new, do not ask when it started. Phrases like "has been like that for a while now" answer **both** the start time and the recurrence question — treat both as answered.
   → Only ask this question if there is **zero** time context in the message and the issue could have just started moments ago.

5. **Are you working from home or the office?**
   → If they are working from the office, also ask: *Which office location are you at?*
   → SKIP if the customer already mentioned their work location in any form (e.g. "I'm at home", "working remotely", "here at the downtown office").
   → If they mention working remotely or from home, no follow-up location question is needed.

6. **Can you please attach any screenshots or photos of the error you are seeing?**
   → Ask this question **only** if the ticket is classified as an **Incident** (something broken that was previously working).
   → **NEVER** ask this for Requests (new user onboarding, equipment requests, access provisioning, software installs, or anything that is new and not broken).
   → SKIP if the customer has already attached screenshots or described the exact error message text in detail.

---

## Step 3 – Compose the Response

**ONLY** if there is at least one genuinely unanswered item, send **ONE** message using the exact greeting below, followed by **ONLY** the questions that were not answered or inferred:

> "Thank you for contacting RockIT Support. To help assist with your request as quickly as possible, can you please provide the following information?"

`[List ONLY the unanswered questions here]`

Then, always append the following at the end of every intake message, after all questions:

> 💬 **Did you know?** For the fastest support and resolution, you can submit and follow up on your tickets directly through Chat in Microsoft Teams! Just look for the IT Help icon in the navigation bar of your Teams app — it's the quickest way to get help from our team.

---

## Critical Rules

- First-person messages ("I", "my", "me") imply the individual user is affected — do **not** ask "who is affected?" unless others may also be impacted.
- Any mention of a time frame, duration, or frequency answers question 4 — do **not** ask when the issue started if any time context exists in the message.
- Any named device, system, app, or technology answers question 3 — do **not** ask what is affected if the customer described their issue with any specificity.
- When in doubt about whether something has been answered, assume it **HAS** been and do **not** ask.
- **ALL** missing questions must be combined into **ONE** single message.
- If **ALL** applicable items are answered or inferable, do **not** send this intake message — proceed directly with triage.
- Question 6 (screenshots) is only ever asked for Incidents — never for Requests, regardless of any other condition.
- Do **not** apply this rule to Thread Messenger tickets.
